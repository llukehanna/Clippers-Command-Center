# Phase 16.1: Bug Fixes - Research

**Researched:** 2026-03-23
**Domain:** Next.js App Router — UI layout, API data shape, DB query correctness
**Confidence:** HIGH (all findings based on direct source code inspection)

## Summary

Phase 16.1 addresses five post-launch bugs identified from user observation. All five bugs have been traced to specific root causes through direct code inspection. No new libraries are needed; every fix is a targeted change to existing code.

Three bugs are pure UI/layout issues (nav centering, live stats display, betting columns). One is a data-currency issue requiring a DB operation (historical game statuses). One is a data-population issue requiring DB investigation (player stats). All five have clear, self-contained fixes.

**Primary recommendation:** Fix each bug independently in separate plans. None of the bugs is systemic — each has a single identifiable root cause with a minimal targeted fix.

## Bug-by-Bug Root Cause Analysis

### Bug 1: Nav Bar Not Centered on Non-Live Pages

**Root cause:** Two entirely separate nav implementations exist.

The live page (`/app/live/page.tsx`) renders its own `LiveFloatingNav` component (lines 16–46) — a centered floating pill with `mx-auto` and `max-w-max`. This is the "correct" centered design.

`TopNav` (`/components/nav/TopNav.tsx`) is the nav for all OTHER pages. It explicitly hides itself on `/live` paths (line 11: `if (pathname.startsWith('/live')) return null`). The TopNav renders a `<header>` with `fixed top-0 left-0 right-0` and `flex items-center gap-6 px-6` — left-aligned, not centered.

**The fix:** Replace the TopNav implementation with the same centered floating pill pattern used in `LiveFloatingNav`. The `LiveFloatingNav` component exists inside `live/page.tsx` but should be extracted and shared. Alternatively, rewrite `TopNav` to use `mx-auto` centering with the same `Surface variant="nav"` and `NavLinkCapsule` components.

**Files involved:**
- `/components/nav/TopNav.tsx` — needs rewrite to centered pill layout
- `/app/live/page.tsx` — `LiveFloatingNav` can be extracted to a shared component
- `/app/layout.tsx` — TopNav is rendered here; may need `pt-14` adjusted if nav becomes floating

**Confidence:** HIGH

### Bug 2: Live Display Missing Stats (Only Box Score and FT Edge Shown)

**Root cause:** The `KeyMetricsRow` component (`/components/live/KeyMetricsRow.tsx`) only renders what the `/api/live` route returns in `key_metrics` — which is limited to 4 metrics: `efg_pct`, `tov_margin`, `reb_margin`, `pace`, plus a client-computed `FT Edge`. This is working correctly.

The actual complaint is that the live page shows "only box score and FT edge" — meaning the KeyMetricsRow is either empty (no live snapshot in DB) or the data state is `DATA_DELAYED` / `NO_ACTIVE_GAME`.

Looking at the `DATA_DELAYED` path in `/app/api/live/route.ts` (lines 306–321): when stale, it returns `key_metrics: []` and `box_score: null`. The live page (`/app/live/page.tsx` line 128) renders `KeyMetricsRow` with `metrics={data?.key_metrics}` — when empty array, it would show an empty grid, not skeletons.

The more likely scenario: there is no active game right now, so `state === 'NO_ACTIVE_GAME'` and the `NoGameIdleState` component renders. The user may have been viewing the page during a game where the poll daemon was offline.

**However**, there is a structural gap: when the poll daemon is running during a live game, the `buildTeamTotals` function in `/app/api/live/route.ts` (lines 208–218) only returns `PTS, REB, AST, TO, FG, 3PT, FT` — it omits `STL, BLK, MIN, +/-` from totals. The `BoxScoreModule` column definition includes all 12 columns including STL/BLK. Player rows do include all stats (line 188–205 in `buildPlayerRow`). Totals row will show `—` for STL, BLK, MIN, +/-.

For the `key_metrics` row: the API returns exactly 4 metrics (efg, tov, reb, pace). The component adds FT edge as a 5th. This is the full spec (LIVE-05). If user expects "shots, rebounds, assists, turnovers" those ARE in the box score already.

**Most likely interpretation:** The user is seeing the `DATA_DELAYED` state which returns `box_score: null` and `key_metrics: []`, resulting in skeleton placeholders with no real data, and only FT edge renders (computed client-side from `0-0` FT strings giving `+0`).

**Fix options:**
1. For `DATA_DELAYED`: the API currently returns `box_score: null`. It could instead return the box score data from the snapshot payload even when stale — the data is there in `payload.home_box` / `payload.away_box`.
2. Confirm with user whether this is a "no game happening" situation or a stale data situation.

**Files involved:**
- `/app/api/live/route.ts` — `DATA_DELAYED` branch (lines 305–321) returns `box_score: null`; could build box score from stale snapshot

**Confidence:** MEDIUM (interpretation of "missing stats" is ambiguous — likely DATA_DELAYED showing nulls)

### Bug 3: Betting Columns Empty in Schedule View

**Root cause:** The `ScheduleTable` component correctly reads `g.odds?.spread`, `g.odds?.moneyline`, `g.odds?.over_under` — but the `OddsSnapshot` type (from `/src/lib/types/odds.ts`) uses different field names: `spread_home`, `spread_away`, `moneyline_home`, `moneyline_away`, `total_points`.

The `ScheduleTable` (`/components/home/ScheduleTable.tsx` lines 55–57) reads:
```
spread: g.odds?.spread ?? '—',
ml: g.odds?.moneyline ?? '—',
ou: g.odds?.over_under ?? '—',
```

But the `getLatestOdds()` function returns an `OddsSnapshot` with fields `spread_home`, `spread_away`, `moneyline_home`, `moneyline_away`, `total_points` — not `spread`, `moneyline`, or `over_under`. These field names don't exist on the returned object, so they are always `undefined`, which falls back to `'—'`.

The API routes pass the raw `OddsSnapshot` result directly into the odds field without transforming it to the field names the UI expects.

**Fix:** Either:
1. Transform odds in the API route (`/src/app/api/schedule/route.ts` and `/src/app/api/home/route.ts`) to map `spread_home`→`spread`, `moneyline_home`→`moneyline`, `total_points`→`over_under`
2. Or update `ScheduleTable` to read the correct field names from `OddsSnapshot`

Option 1 is better — keeps the API shape stable and intuitive for consumers.

**Secondary concern:** The odds sync job (GitHub Actions) runs the `sync-odds.ts` script nightly. The `odds_snapshots` table may be empty or stale if the ODDS_API_KEY is not set or the provider has no games. Even if field names are fixed, if `getLatestOdds` returns null for all games, odds will still show `—`. This is correct behavior per SCHED-04 (show null, never fabricate).

**Files involved:**
- `/src/app/api/schedule/route.ts` — needs odds field mapping (lines 92–120)
- `/src/app/api/home/route.ts` — needs same odds field mapping (lines 332–348)
- `/components/home/ScheduleTable.tsx` — reads wrong field names OR API needs to transform

**Confidence:** HIGH (field name mismatch is definitive)

### Bug 4: Historical Explorer Shows Only Games Since March 1st / Games After March 9th Show as "Scheduled"

**Root cause — date filter (March 1st only):** The history page (`/app/history/page.tsx`) fetches from `/api/history/games?season_id=...&limit=200`. The `/api/history/games` route has NO date filter — it returns all games for the season ordered by `game_date DESC`. This should return the full season.

However, the `season_id` being used is derived from `seasons.at(-1)?.season_id`. The seasons endpoint returns `DISTINCT season_id FROM games`. If the 2025-26 season data (season_id=2025) only exists from the schedule sync (which ran from March onward), games from October–February 2025 would not be in the DB at all.

The likely scenario: the `games` table for season_id=2025 was populated by `sync-schedule.ts` which runs via GitHub Actions (`sync-schedule.yml` daily at 6am PT). If the backfill in Phase 16-03 only fetched upcoming games (future-dated), historical games from earlier in the 2025-26 season (Oct 2025–Feb 2026) may not be in the DB.

From MEMORY.md: "Upcoming schedule: 15 games from Mar 2026 onward, no odds in DB" — this confirms only upcoming games were ingested, not the full season.

**Root cause — March 9th games showing as "Scheduled":** Games that have been played after the backfill date still have `status='scheduled'` in the DB because no finalization job has run for them. The `/api/history/games` route returns `result: null` and `final_score: null` for these games since scores are null in DB. The `GameListTable` shows "SCH" for `isScheduled = g.result === null && !g.final_score`.

**Fix:**
1. **Missing historical games:** Run `npm run sync:schedule` or a custom backfill to populate the full 2025-26 season schedule in the `games` table (all games from October 2025 forward). This is a data ingestion task.
2. **Stale game statuses:** Run `npm run finalize-games` (the finalize-games script from Phase 7) to update completed games from NBA live JSON. This will update `status='final'` and populate box scores for completed games.

Both are operational tasks that require running scripts against the production DB, not code changes.

**Files involved (for code changes, if any):**
- Script: `scripts/sync-schedule.ts` — may need to backfill historical games, not just future ones
- Script: `scripts/finalize-games.ts` — needs to run for games since March 9th
- GitHub Actions `sync-schedule.yml` — may need a one-time backfill flag added

**Confidence:** HIGH (confirmed by MEMORY.md stating only 15 upcoming games from March 2026 are in DB)

### Bug 5: Player Stats Not Populated

**Root cause:** The `/api/players/{id}` route queries `rolling_player_stats` for `trend_summary` and `charts`, and `game_player_box_scores` for `game_log` and `splits`.

From MEMORY.md:
- Game data covers only 2 LAC games (Jan 1 & Jan 8, 2024, season 2023-24)
- `player_team_stints` was manually populated for 11 players
- `is_active=true` for 11 players

The `rolling_player_stats` table is populated by `npm run compute-stats` (Phase 5 script). If this has not been run for the 2025-26 season, or if box score data for the current season has not been finalized, `rolling_player_stats` will have no current-season rows.

The `game_player_box_scores` table is populated by the finalization pipeline (Phase 7 / Phase 16 cron). If games are not finalized (status not 'final'), box scores won't be written.

**Fix:** Same as Bug 4 — finalize games first (populates `game_player_box_scores`), then run `npm run compute-stats` (populates `rolling_player_stats`). These are data pipeline tasks.

Additionally, the `/api/players` route filters by `is_active = true` AND `player_team_stints`. From MEMORY.md: "Post-backfill maintenance: re-set LAC player is_active=true after each npm run backfill run (backfill resets all players to false)." If a recent backfill ran without resetting `is_active`, the player list would be empty.

**Files involved:**
- DB state: `game_player_box_scores` (needs finalization jobs to run)
- DB state: `rolling_player_stats` (needs `npm run compute-stats` after finalization)
- DB state: `players.is_active` (may need manual SQL reset if backfill ran recently)

**Confidence:** HIGH

## Architecture Patterns

### Pattern 1: Centered Floating Nav (existing, used on /live)

The live page implements the correct nav pattern in `LiveFloatingNav`. The fix for Bug 1 is to extract this pattern and apply it to `TopNav`.

```tsx
// Source: /app/live/page.tsx — LiveFloatingNav (lines 16-46)

  <Surface variant="nav" className="relative mx-auto flex max-w-max items-center gap-6 px-6 py-2.5">
    ...
  </Surface>
</header>
```

The `Surface variant="nav"` component is already defined and provides the frosted glass treatment. `NavLinkCapsule` provides the active-state capsule per the design spec.

The non-live `TopNav` uses a different pattern: `fixed top-0 left-0 right-0 h-12 border-b border-border-subtle bg-background/80`. This should be replaced by a floating centered layout. The `-mt-14` on live page content offsets the fixed nav height — this will need to be reconsidered for the shared layout.

### Pattern 2: Odds Field Name Mapping

The `OddsSnapshot` type uses specific field names that differ from what the ScheduleTable reads:

| OddsSnapshot field | What ScheduleTable reads | Fix: map to |
|--------------------|-------------------------|-------------|
| `spread_home` | `spread` | `spread_home` (or map in API) |
| `moneyline_home` | `moneyline` | `moneyline_home` |
| `total_points` | `over_under` | `total_points` |

The cleanest fix is to transform in the API routes before returning to the client, so the API shape is stable and display-friendly. However, since spread has two values (home/away), a decision is needed: show LAC's spread (perspective-adjusted), or show raw `spread_home`.

For display: LAC spread perspective — if LAC is home, show `spread_home`; if LAC is away, show `spread_away`. The `/api/schedule` route already knows `isHome` for each game (line 94).

### Pattern 3: Data Pipeline Sequence

For bugs 4 and 5, the correct repair sequence is:

```
1. npm run sync:schedule (backfill full season game records)
2. Run finalize-games script for each completed game since March 9th
3. npm run compute-stats (recompute rolling_player_stats from new box scores)
4. SQL: UPDATE players SET is_active=true WHERE ... (if backfill reset flags)
```

## Common Pitfalls

### Pitfall 1: Nav Centering Breaks Layout Offset
**What goes wrong:** TopNav is currently `fixed top-0` with `pt-14` on `<main>` in `layout.tsx`. If TopNav changes to floating (not fixed), the `pt-14` offset must be removed or adjusted.
**How to avoid:** When converting to floating nav, update `layout.tsx` to remove the main `pt-14`. The live page already uses `-mt-14` to counteract it — that will also need updating.
**Warning signs:** Content hidden behind nav, or extra blank space at top of page.

### Pitfall 2: Odds Null vs. Empty Object
**What goes wrong:** `getLatestOdds` returns `null` when no odds exist (correct). But if the DB has no rows in `odds_snapshots`, fixing the field names still shows `—` for all games.
**How to avoid:** Verify odds are actually in the DB before concluding the field name fix is sufficient. Check: `SELECT COUNT(*) FROM odds_snapshots;`
**Warning signs:** All odds show `—` even after field name fix.

### Pitfall 3: Historical Season Only Has March+ Games
**What goes wrong:** Fixing `status` for post-March-9 games won't show October–February games if they were never ingested.
**How to avoid:** Run `sync-schedule.ts` with a flag to backfill the full season, not just upcoming games. Alternatively, use the BDL API to backfill historical game metadata.
**Warning signs:** Season game count in history page < 40 (NBA plays 41 home games per season).

### Pitfall 4: DATA_DELAYED Shows No Box Score
**What goes wrong:** The `DATA_DELAYED` branch in `/api/live/route.ts` explicitly returns `box_score: null` and `key_metrics: []`. Even when a recent snapshot exists (< 60s old but poll says stale), the box score won't render.
**How to avoid:** For Bug 2, if fixing DATA_DELAYED: build `boxScore` and `keyMetrics` from the stale snapshot payload using the same `buildBoxScore` and `computeKeyMetrics` helpers — the data is available in `payload.home_box` / `payload.away_box`.
**Warning signs:** User sees skeleton placeholders instead of last-known stats during a game.

### Pitfall 5: Player is_active Reset
**What goes wrong:** `npm run backfill` resets all `players.is_active = false`, causing `/api/players` to return 0 results even for real LAC players.
**How to avoid:** After any backfill run, execute: `UPDATE players SET is_active=true WHERE EXISTS (SELECT 1 FROM player_team_stints pts JOIN teams t ON pts.team_id=t.team_id WHERE pts.player_id=players.player_id AND t.nba_team_id=13);`
**Warning signs:** Roster page shows 0 players.
