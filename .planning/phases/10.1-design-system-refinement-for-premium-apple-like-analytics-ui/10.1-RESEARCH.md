## Summary

Phase 10.1 is a pure visual refinement phase. The full component set exists from Phase 10; this phase changes how everything looks without touching data, APIs, or structure. The entire scope is: update CSS custom property tokens in `globals.css`, update the `@theme inline` block to expose new tokens as Tailwind utilities, and refine each component file to use the new token values and tighter spacing/typography specifications.

The key architectural insight is that Tailwind v4's `@theme inline` mechanism means any new CSS custom property added to `globals.css` automatically becomes available as a Tailwind utility class. Adding `--space-1: 4px` makes `p-[--space-1]` available, or it can be mapped explicitly in `@theme inline`. This means the token update in `globals.css` is the single point of change that propagates to all components.

The existing MASTER.md is stale — it has conflicting values (light background, Fira fonts, blue primary) from a generic design system tool output. This phase replaces it wholesale with the new canonical direction. All Phase 11–14 planning agents will reference the updated MASTER.md.

**Primary recommendation:** Do the work in four discrete steps — (1) update tokens in `globals.css`, (2) refine each component JSX, (3) update MASTER.md, (4) visual review against the reference list. Plan can be structured as one wave covering all five components plus the globals.

### Pattern 2: Tailwind v4 @theme inline extension

**What:** New semantic tokens exposed as Tailwind color utilities in the `@theme inline` block.
**When to use:** For any token that needs a Tailwind utility shorthand (e.g., `bg-chart-1`).

**Example:**
```css
@theme inline {
  /* existing entries stay... */
  --color-chart-1: var(--chart-1);
  --color-chart-2: var(--chart-2);
  --color-chart-3: var(--chart-3);
  --color-chart-4: var(--chart-4);
}
```

After this, `stroke="var(--chart-1)"` in Recharts and `className="bg-chart-1"` in JSX both work.

### Pattern 3: Bypass shadcn Card wrapper for StatCard

**What:** StatCard currently wraps in `<Card>` / `<CardContent>`. The Card primitive applies its own background, border, and padding via CSS variables that may conflict with the new token values. Replacing with a plain `<div>` gives precise control.
**When to use:** When a shadcn primitive is fighting the token values you want.

**Example (StatCard.tsx):**
```tsx
// Source: Tailwind + direct token usage pattern
export function StatCard({ label, value, delta, context, positive, className }: StatCardProps) {
  return (
    <div
      className={cn(
        'rounded-lg border bg-surface px-4 py-3',
        'border-white/[0.06]',
        className
      )}
    >
      <p className="text-[0.6875rem] font-semibold uppercase tracking-[0.08em] text-muted-foreground">
        {label}
      </p>
      <p className="mt-1 text-[1.75rem] font-bold tabular-nums leading-none tracking-tight text-foreground">
        {value}
      </p>
      {/* delta/context row */}
    </div>
  )
}
```

Note: `border-white/[0.06]` is a Tailwind v4 arbitrary opacity syntax — confirmed supported.

**What:** Recharts styling is applied through inline prop values, not CSS classes. Colors, font sizes, and stroke widths are set on the Recharts component props directly.
**When to use:** All chart visual changes.

### Pattern 5: BoxScoreTable — direct HTML table (already raw)

**What:** BoxScoreTable already uses raw HTML `<table>` inside an overflow div — this is ideal. Changes are all in className values.
**When to use:** The existing approach is correct; update class values only.

**Key changes:**
- Outer div: remove `border-border`, keep overflow + maxHeight (no outer card border)
- `<thead>`: `bg-surface border-b border-white/[0.04]` (reduce from `border-border`)
- `<th>`: `text-[0.6875rem] tracking-[0.06em]` (from `text-xs tracking-wide`)
- `<tr>`: `border-b border-white/[0.04] hover:bg-white/[0.03]` (from `border-border hover:bg-muted/40`)
- `<td>`: `text-[0.8125rem] py-0` with `min-h-[36px]` enforced via row class

### Anti-Patterns to Avoid

- **Hardcoding hex values in JSX:** All colors must come from CSS variables. If a token doesn't exist, add it to `globals.css` first.
- **Mixing :root and .dark with different values:** Per Phase 10 decision, `:root` and `.dark` are identical — dark-only design. If you add a token, add it to both blocks with the same value.
- **Using Tailwind color scale (e.g., `slate-800`):** These bypass the token system. Use `bg-surface`, `text-muted-foreground` etc.
- **Over-using `border-b-2` for active states:** Active nav underline is `border-b border-primary` at `1px` — thin, not thick.
- **Adding hover lift to StatCard:** StatCard is display-only. `hover:` effects belong only on interactive elements (nav links, table rows, buttons).
- **Leaving CartesianGrid vertical lines on:** BarChartWrapper already has `vertical={false}` — confirm LineChartWrapper also sets this.
- **Keeping `strokeDasharray="3 3"`:** The new direction uses solid hairlines at very low opacity, not dashed lines.

## Common Pitfalls

### Pitfall 1: Rgba border values not working as Tailwind classes

**What goes wrong:** `border-rgba(255,255,255,0.06)` is not valid Tailwind syntax. Writing `border-white/[0.06]` requires Tailwind to know `white` is a registered color.
**Why it happens:** Tailwind v4's `@theme inline` maps colors. `white` is built-in. The `/[0.06]` arbitrary opacity syntax works with registered colors only.
**How to avoid:** Use `border-white/[0.06]` (Tailwind built-in white + arbitrary opacity) OR use `className="border border-[rgba(255,255,255,0.06)]"` (arbitrary value). Both are valid in Tailwind v4.
**Warning signs:** Purge removes the class or no border appears in dev.

### Pitfall 2: shadcn Skeleton component background

**What goes wrong:** The shadcn `<Skeleton>` component has its own background defined via `--muted` token. If you just add a className override, specificity may not win.
**Why it happens:** shadcn Skeleton uses `bg-muted` internally, which maps to `--muted`. Setting `className="bg-white/[0.05]"` may be overridden.
**How to avoid:** Two options: (a) update `--muted` token value directly in `globals.css` to `rgba(255,255,255,0.05)`, or (b) use `!bg-white/[0.05]` (Tailwind important modifier). Option (a) is cleaner — update the `--muted` token since it aligns with the new surface palette.
**Warning signs:** Skeleton shimmer is too bright/visible against the new darker background.

### Pitfall 3: BoxScoreTable outer container border removal

**What goes wrong:** Removing the outer border from BoxScoreTable's container div means the scrollable area has no visual boundary. On darker backgrounds this creates unclear visual containment.
**Why it happens:** The CONTEXT.md says "No outer card border — table floats with section context." This is intentional but requires the section heading and spacing to do the visual grouping work.
**How to avoid:** When implementing BoxScoreTable, rely on the section label above (using `--text-section-title` scale) and page grid gap to separate it visually. Don't add the border back — let the planner/implementer know this is intentional.
**Warning signs:** Table appears to float without context — this is correct behavior, the surrounding page layout provides the frame.

### Pitfall 4: Typography custom properties not applied via Tailwind

**What goes wrong:** The CONTEXT.md defines tokens like `--text-page-title` with full shorthand values (`1.5rem / 2rem, weight 600, tracking`). CSS custom properties can't hold compound values and be applied as a single utility class.
**Why it happens:** CSS `var()` resolves to a single value. Font-size, line-height, weight, and letter-spacing are separate CSS properties.
**How to avoid:** Define `--text-page-title` as just the font-size value (e.g., `1.5rem`). Apply weight, tracking, and line-height as separate Tailwind utilities. OR add them to `@theme inline` as individual size tokens and use conventional Tailwind classes (`text-2xl font-semibold tracking-tight`). The CONTEXT.md defines these for documentation reference, not as single-apply CSS variables. Each property must be set individually in JSX.
**Warning signs:** Attempting `@apply var(--text-page-title)` — this is not valid CSS.

### Pitfall 5: Border token change affects all shadcn components

**What goes wrong:** The `--border` token is used by shadcn as the default border for ALL components (inputs, selects, etc.) via `@apply border-border` in `@layer base`. Changing it to `rgba(255,255,255,0.06)` from `#334155` affects every element.
**Why it happens:** Phase 10 set up `@layer base { * { @apply border-border; } }`. This is global.
**How to avoid:** This is the intended behavior — the new hairline border IS the desired default. The concern is inputs and other UI elements that need slightly more visible borders. If inputs need more contrast, override locally with `border-white/[0.12]` on those specific components. Accept the global change and adjust outliers.
**Warning signs:** Form inputs (if any appear in later phases) look borderless — add per-component overrides.

### Pitfall 6: Recharts strokeOpacity vs opacity prop

**What goes wrong:** `opacity` prop on Recharts `<CartesianGrid>` applies to the entire element including any stroke. `strokeOpacity` is the correct prop for controlling just the grid line transparency.
**Why it happens:** Common confusion between SVG `opacity` (full element) and `stroke-opacity` (stroke only).
**How to avoid:** Use `strokeOpacity={0.08}` not `opacity={0.08}` on CartesianGrid. The current code already uses `opacity={0.5}` — this should be replaced with `strokeOpacity={0.08}`.
**Warning signs:** Grid lines disappear entirely or affect other chart elements unexpectedly.

## Open Questions

1. **`--radius` global change impact on shadcn components**
   - What we know: Current `--radius: 0.75rem`. CONTEXT.md specifies StatCard radius `8px` (0.5rem). The `--radius` token drives shadcn's `--radius-sm/md/lg/xl` calculations.
   - What's unclear: Changing `--radius` from `0.75rem` to `0.5rem` affects all shadcn components globally. This may or may not be desired.
   - Recommendation: Set `--radius: 0.5rem` for the slightly-tighter-than-current feel (StatCard becomes 8px, other components scale proportionally). This is within Claude's discretion.

2. **Spacing tokens in @theme inline vs CSS-only**
   - What we know: `--space-N` tokens are defined in CONTEXT.md. Tailwind utilities like `p-4` (16px) are already in use.
   - What's unclear: Whether the spacing tokens need to be exposed as Tailwind utilities or can remain as CSS variables for direct use in inline styles.
   - Recommendation: Add `--space-*` tokens to `globals.css` `:root` / `.dark` blocks for documentation and direct `var()` usage. Do NOT map them through `@theme inline` (this would conflict with Tailwind's own spacing scale). Use existing Tailwind spacing utilities (`p-3`, `p-4`, `gap-4`) which already align with the 4px base. The CONTEXT.md spacing values map directly to Tailwind default scale: `space-3=12px=Tailwind p-3`, `space-4=16px=Tailwind p-4`, etc.

3. **Chart wrapper palette prop**
   - What we know: CONTEXT.md leaves to Claude's discretion whether chart wrappers accept a `palette` prop.
   - What's unclear: Phase 11 and 12 chart usage patterns are not yet known.
   - Recommendation: Default to the 4 chart tokens with no `palette` prop for Phase 10.1. The series `color` prop already accepts any string — callers can pass `var(--chart-1)` etc. This maintains the existing API and avoids over-engineering.
