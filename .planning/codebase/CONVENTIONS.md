## Naming Patterns

**Files:**
- Next.js App Router file conventions: `page.tsx`, `layout.tsx`, `route.ts`, `loading.tsx`, `error.tsx`
- API routes located at `src/app/api/.../route.ts` (specified in `Docs/API_SPEC.md`)
- Job/service files: `snake_case` matching job names defined in `Docs/INGESTION_PLAN.md` (e.g., `sync_reference_data.ts`, `poll_live_clippers_game.ts`)
- Utility/helper files: `camelCase.ts` or `kebab-case.ts` — choose one and apply consistently

**Functions and Variables:**
- Use `camelCase` for all functions, variables, and object properties
- Use `PascalCase` for React components, TypeScript interfaces, and type aliases
- Database column names use `snake_case` (as defined in `Docs/DB_SCHEMA.sql`)
- API JSON response keys use `snake_case` throughout (as specified in `Docs/API_SPEC.md`)

**TypeScript Types:**
- Prefix interfaces with no `I` prefix — use plain descriptive names: `GameState`, `InsightPayload`, `LiveApiResponse`
- Use `type` for union types and simple aliases; `interface` for object shapes that may be extended
- Enum-like string unions preferred over TypeScript enums: `"live" | "between_games" | "historical"` (matches DB schema)

**Database Identifiers:**
- Internal IDs: `game_id`, `player_id`, `team_id` (BIGSERIAL)
- Provider IDs: `nba_game_id`, `nba_player_id`, `nba_team_id` — always store both, never conflate
- UUIDs: `insight_id` (UUID, generated by `gen_random_uuid()`)

## API Response Conventions

All API endpoints return `application/json; charset=utf-8` and must include a `meta` envelope:

```typescript
interface ApiMeta {
  generated_at: string;       // ISO 8601 UTC
  source: "nba_live" | "balldontlie" | "db" | "odds_provider" | "mixed";
  stale: boolean;
  stale_reason: string | null;
  ttl_seconds: number | null;
}
```

Error responses use a consistent envelope (non-2xx only):

```typescript
interface ApiError {
  error: {
    code: "BAD_REQUEST" | "NOT_FOUND" | "RATE_LIMITED" | "UPSTREAM_UNAVAILABLE" | "DATA_NOT_READY" | "INTERNAL_ERROR";
    message: string;
    details?: Record<string, unknown>;
  }
}
```

**Null vs. omit:** Missing optional fields must return `null` explicitly rather than being omitted. Example: `"odds": null` when no odds available. Never fabricate or estimate missing values.

**Timestamps:** All timestamps are ISO 8601 strings in UTC. Example: `"2026-03-06T01:23:45Z"`.

**Pagination:** Lists use `cursor`/`limit` pagination; responses include `next_cursor: string | null`.

## State Modeling

Live game state is represented as a discriminated string union:

```typescript
type GameState = "LIVE" | "NO_ACTIVE_GAME" | "DATA_DELAYED";
```

- `LIVE`: active Clippers game in progress
- `NO_ACTIVE_GAME`: no Clippers game active
- `DATA_DELAYED`: upstream failing; serving last cached snapshot

Game status (in DB and API) uses: `"scheduled" | "in_progress" | "final"`.

Insight scope uses: `"live" | "between_games" | "historical"`.

---

## Data Integrity Rules

These rules are non-negotiable and must be enforced at every layer:

1. **Never fabricate facts.** If data is missing, return `null` or empty arrays — never estimate or invent.
2. **Insights require proof.** Every insight returned to UI must include `proof.summary` and `proof.result`. Insights without proof payloads must not be returned.
3. **Provider IDs must be stored alongside internal IDs.** Never lose the upstream ID for games, players, or teams.
4. **Do not overwrite good data with null on provider failure.** Jobs must preserve last known good state.
5. **Odds are append-only.** `odds_snapshots` rows are never updated; UI always reads the latest by `captured_at DESC`.

## Error Handling

**API Routes:**
- Catch all errors at the route handler level
- Return appropriate error envelope with descriptive `code` and `message`
- On upstream failure, return `state: "DATA_DELAYED"` with last cached data (not a 5xx error)
- On missing optional data: return `null` for that field, not an error

**Ingestion Jobs:**
- Jobs must be idempotent — reruns must not create duplicates
- Per-game operations must use database transaction boundaries
- On partial failure, log which game IDs failed and continue processing others
- Never delete existing good data because an upstream call failed
- Exponential backoff for live polling on repeated failures (ceiling ~60 seconds)

**Database:**
- Use upserts (`INSERT ... ON CONFLICT ... DO UPDATE`) for all ingestion paths
- Transactions required for any write spanning multiple tables (e.g., finalize game + box scores)
- Progress checkpoints stored in `app_kv` table for resumable operations

## Caching Strategy

Implemented via HTTP cache headers on Next.js API routes:

| Endpoint | Cache-Control | Notes |
|---|---|---|
| `/api/live` | `no-store` | May use 2–5s in-memory cache for concurrent requests |
| `/api/home` | `public, max-age=300` | 5 minutes |
| `/api/schedule` | `public, max-age=1800` | 30 minutes |
| `/api/history/*` | `public, max-age=86400` | Historical data is stable |
| `/api/players` | `public, max-age=3600` | Roster changes infrequently |

## Ingestion Job Conventions

Job names use `snake_case` matching the inventory in `Docs/INGESTION_PLAN.md`:
- `sync_reference_data`
- `sync_schedule`
- `poll_live_clippers_game`
- `finalize_completed_games`
- `sync_odds`
- `compute_advanced_stats`
- `compute_rolling_windows`
- `generate_insights_batch`
- `backfill_historical_seasons`
- `healthcheck_data_freshness`

Each job must:
- Be idempotent
- Write a success timestamp to `app_kv` on completion (e.g., `schedule_sync:last_success_at`)
- Log failures with enough detail to debug

Backfill checkpoints stored in `app_kv`:
- `historical_backfill:last_cursor`
- `historical_backfill:last_completed_season`
- `live_poll:last_snapshot_at:{game_id}`

## UI / Component Conventions

From `design-system/clippers-command-center/MASTER.md`:

**Icon usage:** SVG icons only — use Heroicons or Lucide. No emoji as icons.

**Clickable elements:** All must have `cursor: pointer`.

**Transitions:** All hover states and state changes must use transitions in the range 150–300ms.

**Focus states:** Must be visible for keyboard navigation on all interactive elements.

**Motion:** Respect `prefers-reduced-motion`. Live updates must not cause layout shift.

**Insight tile rotation:** Must not shift surrounding layout when rotating between insights.

**Font:** Inter with tabular numerals (`font-feature-settings: "tnum" 1, "lnum" 1;`) for numeric/stat display.

**Theme:** Dark mode default. Background `#0F172A`, surface `#111827`.

**Design tokens** (CSS custom properties — never hardcode values):
- Colors: `--color-primary`, `--color-secondary`, `--color-highlight`, `--color-background`, `--color-surface`, `--color-surface-alt`, `--color-border`, `--color-text-primary`, `--color-text-secondary`, `--color-positive`, `--color-negative`, `--color-neutral`
- Spacing: `--space-xs` through `--space-3xl`
- Type: `--text-xs` through `--text-3xl`
- Shadows: `--shadow-sm`, `--shadow-md`, `--shadow-lg`
- Radius: `--radius-sm`, `--radius-md`, `--radius-lg`

**Anti-patterns (forbidden):**
- Landing page / hero / webinar / conversion-oriented layouts
- Light default background for dashboard pages
- Monospaced fonts for headings or body copy
- Oversized cards with wasted vertical space
- Excessive motion on live updates
- Too many bright accent colors in charts and tables
- Auto-play video

## Performance Targets

From `Docs/REQUIREMENTS.md` and `Docs/MVP_CHECKLIST.md`:

- `/api/live` response: < 200ms
- `/api/home` response: < 300ms
- `/api/history/*` responses: < 400ms
- Database dashboard queries: < 200ms
- Live updates visible on UI within 3 seconds of polling

**Avoid at the database layer:**
- Full table scans on large tables (`game_player_box_scores`, `live_snapshots`)
- Repeated heavy aggregations at query time — use `rolling_team_stats` and `rolling_player_stats` instead

## Comments

- Add comments for non-obvious logic: proof validation, backoff strategy, idempotency boundaries
- Document any deviation from the data integrity rules with rationale
- SQL proof queries stored in `insights.proof_sql` should include a brief comment explaining the claim they verify

## Module Design

**API routes:** One route file per endpoint grouping. Each route handler should call a service/query function — do not put database queries directly in the route handler.

**Jobs:** One file per job. Jobs are invoked by scheduler (GitHub Actions cron or app scheduler).

**Exports:** Prefer named exports over default exports for utility functions and services. Default export acceptable for React page components (Next.js convention).

**Barrel files:** Avoid deep barrel index files that obscure where code lives. Flat, explicit imports preferred.
