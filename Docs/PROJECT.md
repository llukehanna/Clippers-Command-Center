# PROJECT.md

## Product
Clippers Command Center

## Product Definition
Clippers Command Center is a desktop analytics dashboard designed for avid Los Angeles Clippers fans who want deep statistical context while watching games and between games.

The application has three primary modes:

1. **Live Game Mode (Primary Mode)**
   Activated automatically when a Clippers game is in progress. The dashboard continuously displays live game data, player and team performance metrics, contextual statistics from recent games and the current season, and algorithmically generated insights tied to the current game state.

2. **Between‑Games Dashboard (Default Mode)**
   When no Clippers game is active, the dashboard shows broader insights about the team, upcoming schedule information, recent performance trends, and notable contextual statistics about the Clippers.

3. **Historical Exploration Mode**
   Allows exploration of past seasons with an initial focus on player trends (rolling averages, splits, and game‑log‑based trend charts) and historical Clippers context stored in the system database.

Live Game Mode is the **core experience** and receives highest priority in system design and feature development.

---

## Product Principles

The system must follow these guiding rules:

1. **Clippers‑First Context**
   All insights prioritize Clippers performance and events. League data is used only when it provides context for Clippers performance.

2. **Provable Insights Only**
   Every generated insight must be verifiably correct using data stored in the application database.

3. **Information Density Over Visual Decoration**
   The interface prioritizes meaningful data over aesthetic embellishment.

4. **Real‑Time Clarity During Games**
   During live games the interface must update quickly and clearly without disruptive UI changes.

5. **No False or Unverified Facts**
   The system must avoid displaying insights that cannot be proven from stored data.

---

## Definition of “Provable Insight”

A fact or insight is considered provable if it can be verified using stored historical data within the application database.

Example:

"This is the first time in franchise history the Clippers have scored 50 points in a quarter."

To support this claim the system must be able to verify:

• the current game event
• historical franchise data
• the absence of previous occurrences

Each generated insight must store:

- the database query used to verify the claim
- the timestamp when the insight was generated
- the referenced game or season context

If a claim cannot be verified with stored data, the system must **not display the insight**.

## Target User

The primary user is an **avid Los Angeles Clippers fan** who is also knowledgeable about the NBA and interested in statistical context.

The current implementation is intended for personal use but the architecture must allow future expansion to a wider audience.

No user accounts or authentication are required in the initial version.

## Target Platform

The initial version targets:

**Desktop / laptop screens only.**

Mobile optimization is intentionally deferred to later versions.

## Core Use Case

A user watching a Clippers game opens the dashboard on a laptop or desktop display while the game is on television.

The dashboard displays:

• the live scoreboard
• a full box score for the game
• contextual team and player statistics
• recent team performance trends
• upcoming schedule information
• algorithmically generated insights tied to the game

The interface also displays smaller panels showing other NBA games with:

• score
• game clock / time remaining (when available)
• Clippers‑relevant highlights (e.g., standings/seeding implications, direct rivals, opponent scouting relevance)

When no game is active, the dashboard transitions to Between‑Games Mode which shows broader Clippers insights, schedule context, and player trend summaries.

---

## Insight Engine

Insights are generated by a deterministic algorithm that analyzes:

• live game events
• player performance
• team performance
• recent game history
• season‑level statistics
• historical franchise data

The algorithm evaluates candidate insights and displays only those with the highest significance.

Examples of insights:

• scoring runs
• clutch performance indicators
• historical milestones
• rare statistical combinations
• player streaks
• notable comparisons to past Clippers seasons

The system may display multiple insights but must prioritize those most relevant to the current game situation.

### Insight Evaluation Cadence (Live Games)

During Live Game Mode, the insight engine evaluates candidate insights:

- on scoring events when detectable from the live feed
- otherwise at least once every 60 seconds as a fallback

The UI displays **2–3 insight tiles** at a time and rotates them.

---

## Live Data Behavior

The system polls the NBA live data endpoint approximately every **12 seconds** while a Clippers game is active.

During live games, the system stores a snapshot for each poll cycle (i.e., roughly every 12 seconds) to support time‑based trend detection (runs, momentum shifts, clutch windows) and to enable post‑game reconstruction.

Live updates are used to detect:

• scoring runs
• lead changes
• clutch situations
• turnover bursts
• foul trouble
• notable statistical milestones

When no Clippers game is active, the system checks for finalized box scores approximately every **6 hours** to update the historical database.

After each completed game the system retrieves the **final box score** and stores the data permanently.

### Rate‑Limit and Data Quality Handling

- If live endpoints rate‑limit or fail, the system must back off (exponential backoff) and continue serving the most recent cached snapshot to the UI.
- If the system cannot verify an insight from stored data, it must not display it.

## Vegas Odds (Version 1)

The dashboard displays Vegas odds as a context signal (display‑only) and may use them to annotate insights.

- Data source: an external odds feed.
- Behavior: if odds are unavailable, the UI hides odds rather than guessing.
- No betting functionality is included.

## System Architecture

### Frontend

Framework: Next.js
Language: TypeScript
Styling: Tailwind CSS
Component Library: shadcn/ui
Charts: Recharts

### Backend

Next.js server routes provide API endpoints for:

• live game data
• historical stats
• insights
• team and player metrics

### Database

PostgreSQL (hosted on a free serverless provider such as Neon).

PostgreSQL is used instead of SQLite to allow future scalability while remaining free for development.

### Core Tables

- games
- team_box_scores
- player_box_scores
- advanced_stats
- live_snapshots
- generated_insights

### Data Sources

Live data:

NBA public live JSON endpoints

Historical stats:

BALLDONTLIE API

Future expansion:

Basketball Reference scraping

---

## Data Ingestion Strategy

Historical data ingestion and live polling run using scheduled jobs.

Recommended infrastructure:

• Vercel for application hosting
• GitHub Actions scheduled workflows for periodic ingestion tasks

These solutions allow the system to operate entirely within free infrastructure limits.

---

## Long‑Term Direction

Although the initial version is built for a single user, the architecture should allow future expansion to:

• multi‑team dashboards
• public access
• larger historical datasets
• advanced statistical models

The core design goal remains building the **best possible Clippers analytics dashboard for a serious fan.**
